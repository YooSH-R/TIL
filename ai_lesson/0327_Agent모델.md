# Mastering AI Agent
## 1. Introduction to AI Agent
- 에이전트: 다른 사람 또는 단체를 대신하여 행동하는 개인 또는 단체
- 에이전트는 특정 작업을 수행하고, 결정을 내리며, 합의를 도출하는 것을 목표로 함
- AI 에이전트: AI를 사용하여 사용자를 대신해 목표를 추구하고 작업을 수행하는 소프트웨어 시스템
- AI에이전트의 주요 특성: 자율성, 동작(도구), 학습(적응), 추론(계획), 기억, 인식

## 2. Multi Agent System
### 2-1. Multi Agent System
- 지능형 에이전트들이 팀을 이루어 협력하고 지식을 나누며, 함께 문제를 해결하는 능력에 초점을 맞춤
- 구성 요소
  - 에이전트
  - 환경
  - 상호작용
  - 조직

### 2-2. 방법론: 협업 유형
- 협업: 에이전트들은 각자의 개별 목표를 공동의 집단 목표와 일치시켜, 상호 이익이 되는 결과를 달성하기 위해 협력
- 경쟁: 에이전트들은 각자의 개별 목표를 우선시하며, 이는 다른 에이전트들의 목표와 충돌하거나 대립할 수 있는 경쟁 요소를 도입

### 2-3. 방법론: 협업 전략
- 규칙 기반 프로토콜
  - 에이전트 간 상호작용은 미리 정의된 규칙에 의해 엄격히 통제
- 역할 기반 프로토콜
  - 세분화된 목표 하에서 각 에이전트가 고유한 역할 분할을 통해 작동하도록 함

### 2-4. 방법론: 커뮤니케이션 구조
- 중앙집중형 구조
  - 참여자-서비스 제공자 구조
  - 중앙 에이전트
- 탈중앙화 구조
  - 에이전트에 대한 통제 및 의사결정 분산

## 3. Memory & Tool in Multi-Agent System
- 패러다임의 전환: 언어 모델에서 에이전트 시스템으로
- 메모리 & 도구
  - LLM 내장 매개변수 메모리의 한계: 낡은 지식, 환각 현상, 사적 정보 접근 불가

- 검색 증강 생성(Retrieval-Augmented Generation, RAG)
  - LLM이 답변 생성 전 신뢰할 수 있는 외부 소스에서 관련 정보를 검색하고, 해당 정보를 바탕으로 응답 구성
  - 한계: 외부 지식 소스를 하나만 사용, 정보 품질 검토나 확인 없음, 상태 정보 저장하지 않아 여러 단계 추론이나 복합적 탐구 한계

- RAG + 에이전트 = 에이전트형 RAG
  - AI 에이전트가 RAG 파이프라인을 향상시켜, 각 단계들을 조율하고 단순한 검색/생성을 넘어서는 역할 하도록 함
  - 구성 요소: LLM, 메모리, 계획, 도구
  - 단일 에이전트형 RAG, 멀티 에이전트형 RAG 시스템

- 멀티 에이전트 시스템을 위한 훈련 방법
  - Toolformer: 언어모델이 스스로 도구 사용법을 학습
  - Gorilla: LLM을 대규모 API와 연결
  - ReTool: LLM의 전략적 도구 사용을 위한 강화학습
  - Search-R1: LLM을 사용한 더 나은 웹 검색을 위한 강화 학습

- 모델 컨텍스트 프로토콜(MCP)
  - AI 모델이 외부 도구, 리소스, 환경과 원활하게 상호작용할 수 있도록 표준화된 인터페이스

- Agent2Agent 프로토콜
  - 여러 AI에이전트가 내부 메모리, 사고, 도구를 직접 공유하지 않고, 맥락, 작업 업데이트, 지시사항, 데이터만 교환하여 협업할 수 있도록 함

## 4. Reasoning and Planning in AI Agent
### 4-1. LLM의 추론 능력(Reasoning)
- 추론 능력이 있는 LLM
  - 논리 퍼즐, 수학적 문제, 코딩과 같은 복잡한 작업 해결 가능

- LLM의 추론 유도
  - 프롬프트 만으로 추론 능력 이끌어낼 수 있음
    - 1. Few-shot prompting(CoT Prompting)
    - 2. Zero-shot prompting

- LLM의 추론 능력 강화
  - 자기 일관성: 다양한 추론 과정을 생성하고 그 중 가장 일관성있는 답변을 고르는 디코딩 방식
  - Tree of Thoughts: BFS 또는 DFS 방식으로 다양한 생각을 생성하고 평가

### 4-2. LLM 에이전트의 계획 수립(Planning)
- LLM 에이전트의 Planning
  - LLM은 여러 사고 단계를 생성할 수 있음. 즉, 목표 달성을 위한 하위 목표들의 순서를 구성하 수 있음
  - 행동(실행) 명령을 통합하면 LLM이 결과를 되돌아보면서 도구를 상용하거나 행동할 수 있음

- LLM 에이전트의 Planning 개요
  - ReAct: 추론만 하거나 행동만 하는 것이 아니라 '생각'과 '행동'을 함께 생성하는 방식
  - HuggingGPT: 계획 수립을 담당하는 LLM이 복잡한 작업을 나누어 각각의 전문 모델들에게 분배하는 시스템

- LLM 에이전트의 Planning 능력 강화
  - Plan-and-Act: 계획 에이전트는 거시적인 계획을 생성, 실행 에이전트는 해당 계획에 따라 하위 단계 실행

### 4-3. 추론 시간 연산량 최적화 및 효율적 추론 접근법
- 추론 시간의 연산량 증대
  - 추론 시 컴퓨팅 자원 더 많이 투입할 수록 LLM 성능 향상

- 추론 모델
  - 추론을 위한 컴퓨팅 자원이 할당된 추론 모델 등장

- 추론 모델 강화
  - 정확성과 올바른 형식에 대해서만 보상을 제공함으로써 LLM이 CoT를 사용하는 방법 학습하게 함

- 효율적인 추론
  - 과도한 사고 현상
  - 더 긴 CoT 추론 과정은 성능 향상시키지만, 장황하고 중복된 출력으로 인해 계산 부담 늘어나는 것은 문제
  - 모델 기반 접근(가변 길이)
    - DeGRPO: 첫 번째로 생성된 토큰으로 추론 길이를 제어
  - 추론 결과 기반 접근(동적 추론)
    - InftyThink: 일부 추론 과정을 만들어내고, 지금까지의 생각을 요약
  - 입력 프롬프트 기반 접근(라우팅)
    - 쉬운 질문에는 연산량 적게, 복잡한 질문에는 연산량 많이 배정

## 5. Domain-Specific AI Agent
### 5-1. Deep Research의 배경
- 패러다임의 전환: 웹 검색 -> LLM 챗봇 -> 검색 증강 생성(RAG) -> 에이전트 기반 심층 연구
### 5-2. 에이전트 기반 심층 연구
- 핵심 개념
  - LLM은 다단계 추론과 검색 결합
  - 동적 피드백 루프 형성: 검색 결과 추론하고 이 추론은 검색 과정을 조정
  - 간단한 응답 생성기가 아니라 능동적인 정보 탐색 에이전트 역할
  - 필요한 경우 문서를 읽고 쿼리를 반복적으로 생성하고 수정하여 다른 도구나 에이전트와 협력

- 역동적 환경에서의 어려움과 해결방안
  - 검색과 추론의 결합에는 어려움 존재
    - 추론이 언제, 무엇을 검색할지 결정하도록 하여 오류 전파 줄임
  - 적응력 한계
    - 강화 학습
  - 효율적 리소스 할당
    - 작업 유형에 따라 검색/추론 리소스 비율 동적으로 조절하는 기술 필요
  - 많은 요청 동시 처리
    - 분산 CPU 시스템에서 작업 나누어 처리
  - 웹 크롤링 및 API 제약
    - 재시도 메커니즘과 캐시를 사용하여 호출 빈도와 비용 감소
  - 정보 추출 효율성 문제
    - 중요도 떨어진다 판단되면 바로 포기하고 핵심 페이지에 컴퓨팅 자원 집중

- 강화학습 기반 훈련 및 보상 설계
  - 프롬프트
  - 지도학습
  - 강화학습

- Test-Time Scaling(TTS)
  - 추론/검색 단계 수를 늘리면 성능 향상으로 이어짐

- 강화학습 훈련 프레임워크 및 보상
  - 알고리즘: GRPO(Group Relative PPO)
    - 이전 정책을 사용한 여러 그룹의 실행 결과를 기준으로 삼음
    - 새로운 정책이 이 결과들의 발생 확률을 높이는 정도에 따라 PPO 방식으로 업데이트
  - 보상 설계
    - 형식 오류: -1 패널티
    - 올바른 형식: 단어 레벨의 F1 점수(0-1)을 부여
    - 사실에 대한 정확도, 논리적 일관성 등에 대한 복합적 보상을 도입할 수 있음

### 5-4. 평가 및 발전 방향
- 한계점 및 고려사항
  - 높은 비용(토큰)
  - 범용성의 한계
  - 에이전트 관리의 복잡성
  - 상태 관리 및 오류 전파
  - 평가 및 디버깅의 어려움

- 미해결 문제 및 발전 방향
  - 인간 중심의 상호 작용
  - 전문 분야 에이전트
  - 효율적 TTS 구현
